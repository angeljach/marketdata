Introducción

Una vez establecida la comunicación, el Programa Receptor debe hacer una serie de verificaciones de la información que está recibiendo antes de poder pasar esta a los programas aplicativos del usuario.

La información contenida en el Header que acompaña a cada uno de los registros que se envían también debe ser validada, según se muestra más adelante.

Introduction

Once the communication is established, the Receiving Program must make a series of verifications of the information it is receiving before it can be passed to the user´s application programs.

The information contained in the Header that accompanies each of the records that are sent must also be validated, as shown below.

Format of reception of information for the user

<> Information that traveling in binary (the represented values are converted to decimal byte per byte).

<= Symbol to indicate that a line ends.

EXAMPLE:

<128><0><=
000000002170715378902074032110J Q   GFINTER95B-D199512280012280000000010000000<=
0000000100000000000001500000050000000100000000AJ199703280007<=
" " <----- Optional character, only when the frame size is odd.
<199><39>

Record Breakdown:

Byte 1 and 2 Longitude

<148><0> Value of the frame size in binary |10010100|00000000| = 148

Byte 3 and 2 Flags

<128><0> Value in binary |10000000|00000000| = 128
These 2 bytes are used for flags, currently only the first bit of the first byte is occupied and refers to checksum validation.

Byte 5 to 15 Sequence

00000000217

Byte 16 to 23 Time

07153789

Bytes 24 and 25 Type

02

Bytes 26 to 31 Content

074032

Bytes 32 to 34 Longitude

110

Bytes 35 Available

" "

Bytes 36 to 258 Data (Variable record size depends on the type of FORMAT, maximum 961)

0000000100000000000001500000050000000100000000AJ199703280007<=

Byte XXX Filler (Depends on the type of record)

" " <----- Only when the sum of the bytes is odd, to complete the message in par.
NOTE: When using the byte to complete the message in par, add 1 to the length of the frame.
